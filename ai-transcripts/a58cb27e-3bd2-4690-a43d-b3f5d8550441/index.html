Hiya. Here is my briefbot project: https://github.com/shaynelarocque/briefbot

I want to convert this pattern to this takehome assignment I received. My idea is a similar pattern/project but tweaked to hit the goal.

So the briefbot flow is: ingest an application, do a bunch of prelim research with custom search tools, can ask for clarity from the operations staff, and flag things it's not confident about.

My idea is similar for this ReviewLens project, but instead of an application submission it's given a CSV of data or a URL and we scrape (using Firecrawl) for reviews, embed into a vector DB (ChromaDB, in-process), and let an agent run loose on it via a chat interface.

Ingestion: CSV upload should be the polished primary path. URL scraping via Firecrawl is best-effort secondary, review platforms are heavily anti-bot and the assignment already says CSV is perfectly valid. Either way, after ingestion show a clear summary of what was captured (count, date range, rating distribution, platform) so the user trusts the data before they start asking questions.

Chat interface: After upload, the chat greets them: "Hello! I see you wanted to do some analysis on X. Do you have any specific questions about the data?" Below that, generated follow-up prompt buttons, things like "Show me sentiment breakdown", "What are the top complaints?", "Rating trends over time". These regenerate after each response based on what was just discussed.

Agent tools: Give it tools like: - search_reviews — semantic search over the ChromaDB vector store - analyze_sentiment — aspect extraction + sentiment scoring on review subsets - generate_chart — returns Chart.js config JSON that renders inline in the chat (sentiment distribution bars, rating over time lines, aspect breakdowns) - calculate_stats — aggregations, distributions, trend analysis - suggest_follow_ups — contextual next-question buttons

Keep the chart types focused — a few that work really well (bar, line, pie) beats trying to do everything. The agent should decide when a chart adds value, not chart everything.

Scope guard — this is critical. The assignment says "this is one we care a lot about." The AI must ONLY answer questions about the ingested reviews. If a user asks about external platforms, competitors, general knowledge, or anything not in the dataset, it gracefully declines. Three-layer defense: system prompt instructions (primary), retrieval confidence check (if ChromaDB returns nothing relevant, auto-reject), and explicit rejection messaging. Invest real effort here, it's what they'll test first.

Stack: Same as briefbot, FastAPI + Jinja2/HTMX + SSE streaming, Claude Agent SDK with custom MCP tools via create_sdk_mcp_server. Charts via Chart.js embedded in chat messages. Deploy on Render with Docker.

Key difference from briefbot: Briefbot is fire-and-forget (submit → agent runs autonomously → watch the log). ReviewLens is conversational — user sends a message, agent responds with text/charts/follow-ups, user sends another. So the agent loop gets invoked per-message rather than per-session.
