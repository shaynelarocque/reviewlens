ReviewLens AI — Rearchitect to Sandbox Agent Pattern

Context

I'm building ReviewLens AI, a review intelligence portal for an ORM (Online Reputation Management) consultancy. Users upload customer reviews (CSV or URL scrape), and then chat with an AI agent that analyzes those reviews through a guardrailed Q&A interface. It's a take-home assignment for FutureSight.

I have a prior project called BriefBot that used what I call a "sandbox agent" pattern — instead of scripting the agent step-by-step, you give it: knowledge (domain reference material), tools (capabilities it can call in any order), a goal (definition of done), and guidelines (quality bar, not step-by-step instructions). The agent then decides what to do, in what order, and how deep to go.

I want to bring that pattern to ReviewLens, adapted for a conversational agent (per-message invocation with ongoing chat) rather than BriefBot's one-shot autonomous run.

What Exists Now

The app is a working FastAPI + HTMX + SSE app with: - CSV upload with AI-powered column mapping (Claude Haiku maps arbitrary CSV headers to canonical fields) - Optional URL scraping via Firecrawl - ChromaDB vector store for semantic search over reviews - Per-message agent invocation using claude-agent-sdk with a custom MCP server - 5 MCP tools: search_reviews, analyze_sentiment, generate_chart, calculate_stats, suggest_follow_ups - Three-layer scope guard: system prompt + retrieval confidence + architectural boundary (no tools access external data) - SSE streaming with tool activity indicators - Chart.js inline rendering in chat - Follow-up question buttons - Light/dark theme, session history sidebar, sample datasets

What to Change

1. Add a /knowledge folder with ORM domain knowledge

Create a /knowledge directory with markdown files the agent reads contextually. These should contain practical domain knowledge that turns a generic chatbot into a review intelligence analyst. Files like:

  • review-analysis-framework.md — How to approach review analysis for ORM consultancies. Aspect-based sentiment analysis patterns. What constitutes a statistically meaningful finding vs noise. Common review aspects by industry (product quality, customer service, pricing, shipping, etc). How to identify emerging trends vs established patterns. What makes an insight actionable vs merely interesting.

  • analysis-patterns.md — Templates for common analysis types: competitive gap analysis from reviews, churn signal detection, feature request extraction, customer journey pain point mapping, sentiment trend analysis, rating distribution interpretation (what J-curves, bimodal distributions, etc. mean).

  • report-structure.md — How to structure an exportable analysis report. Section templates for executive summary, key findings, aspect breakdown, risk signals, recommendations. Citation standards (always reference specific reviews by quoting/paraphrasing).

Keep these practical and concise — they're reference material for the agent, not documentation for humans. Write them in a tone like an experienced ORM analyst briefing a junior analyst.

2. Create a knowledge.py module (similar to BriefBot's)

Port the pattern from BriefBot's app/knowledge.py — load all markdown files from /knowledge on startup, cache them, and provide get(name) and list_files() functions. The agent should be able to discover and read these via tools.

3. Restructure the system prompt into sandbox quadrants

Rewrite app/prompts.py so build_system_prompt() follows the sandbox structure explicitly:

Knowledge section: Tell the agent it has a reference library and should consult it when relevant. Don't dump the knowledge inline — the agent should use tools to read files, just like BriefBot.

Tools section: Document each tool's purpose and when to use it. Current tools plus the new ones (below).

Goal section: For each message, the agent's goal is: answer the user's question using ONLY the ingested review data, grounded in actual search results, with specific review citations. When a chart would communicate better than text, use one. Always suggest follow-ups.

Guidelines section: The quality bar: - Every claim must be grounded in actual review data from search results. If search returns nothing relevant, say so honestly. - Cite specific reviews — quote or paraphrase actual text with attribution when possible. - Use charts strategically, not reflexively. A chart adds value for distributions, trends, and comparisons. - Be quantitative when possible — counts, percentages, averages. - If the user asks something outside the dataset scope, refuse gracefully and suggest an alternative. - Confidence awareness: if search results are sparse or tangential, acknowledge the limitation rather than overstating findings.

Keep the scope guard — it's the strongest part of the current prompt. The three-layer defense (prompt + retrieval + architecture) is good. Keep the explicit refusal instructions and the "verify before answering" checklist.

Add these tools in app/tools.py alongside the existing ones:

This is NOT conversation memory — it's a deliverable being built alongside the chat.

get_report — Retrieve the current saved report findings. Agent calls this when asked to generate a summary/report, or to check what's already been captured.

check_scope — Explicit scope validation tool. Takes a question string, runs a lightweight check against the dataset metadata (platform, product name, review count). Returns whether the question is in-scope, out-of-scope, or ambiguous. This makes the scope guard tool-backed rather than purely prompt-driven. The agent should call this when a question feels borderline.

Consider summarizing older messages rather than hard-truncating.

6. Add report generation endpoint

Keep it simple — even just a "Generate Report" button that asks the agent to compile saved findings into a structured document would be great.
